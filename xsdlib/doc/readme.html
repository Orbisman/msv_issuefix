The Sun XML Schema Part 2 Java Implemantation (XSDLib) is an implementation of W3C XML Schema Part 2, intended for use by application developers as their building block.

This preview version implements proposed recommendation version of the candidate recommendation. Being a proposed recommendation means it is subject to change. In fact, many important changes are made from 24th,Oct,2000 edition and many changes are going to be made before it becomes a recommendation. So developers should be warned for possible future changes including (but of course not limited to)

  • change/addition/removal of type names
  • change/addition/removal of facets

This product includes software developed by the Apache Software Foundation (http://www.apache.org/)

Known Limitations

  1. For "float" and "double" types. The spec says a lexical value must be mapped to the closest value in the value space, but XSDLib cannot accept values which are bigger than maximum value of that type or smaller than minimum value of that type. However this shouldn't be a problem for the most of users.
  2. "ID", "IDREF", and "IDREFS" types are not implemented. Although uniqueness constraints are removed from part 2, these types are still intended to be used with uniqueness constraints. These constraints are so special that it is impossible to provide a generic implementation. com.sun.tranquilo.datatype.DataTypeFactory does not recognize these three types.
  3. "NOTATION" type is not implemented. com.sun.tranquilo.datatype.DataTypeFactory does not recognize this type name.
  4. "length", "minLength", and "maxLength" facet is effectively limited upto 2147483647. Values above this limit are recognized, but will be treated as this limit. Items longer than this limit will not be validated correctly. Author believes that this limitation has no practical impact.
  5. Implementation of "totalDigits" and "fractionDigits" should be considered as incomplete. The spec is not clear enough.
  6. "anyURI" (former "uriReference") is made to accept IP v6 addresses like "::192.168.0.1", which is not accepted by the original BNF specified in RFC 2373. This modification should be considered as a "bug fix". Although, BNF specified in RFC 2373 has several other problems, those are not fixed. For example, the current release accepts "1:2:3:4:5:6:7:8:9", which is not a valid IP v6 address.
  7. "language" type is implemented in accordance with RFC 1766, and language identifiers are treated in case-insensitive way. XML Schema Part 2 says that the lexical space of language type will be as defined in XML1.0 Recommendation 2nd edition, but that production was thrown away in the 2nd edition. Even worse, the derivation shown in XML Schema Part 2 does not correctly implement the definition given in RFC 1766. So apparently there is a problem in the definition of "language" type. However, by making "language" case-insensitive, it is no longer a derived type of "token" type.
